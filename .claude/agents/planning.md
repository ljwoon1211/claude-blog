# Planning Guide

## 작업 기획 프로세스

> 기능 개발, 환경설정, 리팩토링 등 **모든 작업**에 적용

### 1. 계획 문서 작성

- 위치: `docs/plans/active/{task-name}.md`
- 기존 계획 확인: `docs/plans/completed/` 검토하여 중복/충돌 방지

### 2. 구현

- 백엔드 먼저 (도메인 → 유스케이스 → oRPC) → 프론트엔드 (엔티티 → 피처 → 위젯)
- 보안 체크리스트 확인 (code-conventions.md 참고)

### 3. 완료

- `docs/plans/active/{feature}.md` → `docs/plans/completed/`로 이동
- CHANGELOG.md 업데이트

## 기능 계획 문서 템플릿 (PRD)

```markdown
# {기능명}

## 개요

- **목적**: 이 기능이 해결하는 문제
- **대상 사용자**: 누구를 위한 기능인지
- **우선순위**: P0(필수) / P1(중요) / P2(개선) / P3(희망)

## 요구사항

### 기능 요구사항

- [ ] FR-1: {기능 설명}
- [ ] FR-2: {기능 설명}

### 비기능 요구사항

- [ ] NFR-1: 성능 (응답 시간 < 200ms)
- [ ] NFR-2: 보안 (인증 필요 여부)

## 사용자 스토리

- **AS** {역할}, **I WANT** {행동}, **SO THAT** {가치}

## 기술 설계

### 백엔드

- 도메인: `src/domains/{domain}/`
- oRPC 프로시저: `src/server/orpc/routers/{router}.ts`
- DB 스키마 변경: 있음/없음

### 프론트엔드

- FSD 레이어: entities / features / widgets
- 슬라이스: `src/{layer}/{slice}/`
- 새 컴포넌트 목록

### API 명세

| 프로시저      | 입력                 | 출력   | 인증 |
| ------------- | -------------------- | ------ | ---- |
| `post.create` | `{ title, content }` | `Post` | 필요 |

## 작업 분해 (WBS)

1. [ ] DB 스키마 / 마이그레이션
2. [ ] 도메인 엔티티 / 유스케이스
3. [ ] oRPC 프로시저
4. [ ] 프론트엔드 UI
5. [ ] 테스트
6. [ ] 문서 업데이트

## 검증 기준

- [ ] 모든 유스케이스 단위 테스트 통과
- [ ] E2E 시나리오 통과
- [ ] 보안 체크리스트 확인
```

## 우선순위 기준

| 등급   | 기준                          | 예시                    |
| ------ | ----------------------------- | ----------------------- |
| **P0** | 서비스 핵심, 없으면 출시 불가 | 포스트 CRUD, 인증       |
| **P1** | 사용자 경험에 중요            | 검색, 댓글, 태그        |
| **P2** | 있으면 좋은 개선              | 다크모드, 공유 기능     |
| **P3** | 향후 고려                     | 뉴스레터, 분석 대시보드 |
